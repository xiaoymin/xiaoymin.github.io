通过langchain的上下文压缩器以改进文档检索任务

原文链接地址：Improving Document Retrieval with Contextual Compression

注意：这篇文章假设你对LangChain有一定的了解，并且是有一定的技术能力。

💡 TL;DR：我们引入了一个新的抽象和一个新的文档检索器，以促进检索到的文档进行后置处理。具体地说，新的文档检索器后置处理工作使得获取一组检索到的文档并从中仅提取与给定查询相关的信息变得容易。

简介​

许多LLM驱动的应用程序需要一些可查询的文档存储功能，以允许检索这些尚未融入LLM的特定应用程序的信息。

假设您想创建一个聊天机器人来回答有关您的个人笔记的问题。一种简单的方法是将您的笔记进行段落分片，并将嵌入存储在向量数据库中。当您向系统提问时，它会嵌入您的问题，对向量存储执行相似性搜索，检索最相关的文档（文本块），并将他们组合给到LLM来获取最终得答案。

问题​

这种方法的一个问题是，当您将数据提取到文档存储系统时，您通常不知道将使用哪些特定查询来检索这些文档。在我们的笔记问答示例中，我们只是将文本分成大小相等的块。

这意味着当我们得到一个特定的用户问题并检索文档时，会获取得到该文档有一些相关的文本，但也可能有一些不相关的文本。

在构建LLM大模型的Prompt提示时插入不想关的信息是不好的，因为：

  • 它会分散LLM对这些信息得注意力
  • 占用了插入其他信息的宝贵空间

解决方案​

为了帮助解决这个问题，我们引入了一个DocumentCompressor抽象对象，它允许您在检索到的文档上运行compress_documents(documents: List[Document], query: str) 。

这个想法很简单：我们可以使用给定查询的上下文压缩它们，以便只返回相关信息，而不是立即按原样返回检索到的文档。这里的“压缩”既是指对单个文档的内容进行压缩，又是对文档进行批量过滤。

压缩器的目标是使仅将相关信息传递给LLM。通过这样做，它还使您能够将更多信息传递给 LLM，因为在初始检索步骤中，您可以专注于召回（例如，通过增加返回的文档数量）并让压缩器处理精度。

特性​

我们在 LangChain Python 包中实现了几个新功能：

  • DocumentCompressor抽象对象您可以开箱即用。
  • ContextualCompressionRetriever 将另一个 Retriever 与 DocumentCompressor 一起包装起来，并自动压缩基本 Retriever 的检索文档。

DocumentCompressor一些示例：

  • LLMChainExtractor 使用 LLMChain 从每个文档中仅提取与查询相关的语句。
  • EmbeddingsFilter 嵌入了检索到的文档和查询，并过滤掉嵌入与嵌入查询不够相似的所有文档。这个压缩器本身做的事情与大多数 VectorStore 检索器非常相似，但它作为一个组件变得更有用......
  • DocumentCompressorPipeline ，这使得创建转换和压缩器的管道并按顺序运行它们变得容易。一个简单的例子是您可能想要组合 TextSplitter 和 EmbeddingsFilter 以首先将您的文档分解成更小的部分，然后过滤掉不再相关的拆分文档。

您可以使用任何现有的 Retriever（无论是基于 VectorStore 还是其他方式）尝试这些，例如：

from langchain.llms import OpenAI
from langchain.retrievers import ContextualCompressionRetriever
from langchain.retrievers.document_compressors import LLMChainExtractor

# base_retriever defined somewhere above...

compressor = LLMChainExtractor.from_llm(OpenAI(temperature=0))
compression_retriever = ContextualCompressionRetriever(base_compressor=compressor, base_retriever=retriever)
contextual_retriever.get_relevant_documents("insert query here")
